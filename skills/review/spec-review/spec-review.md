# SPEC Review Generated by AI Agents

Project- and LLM-agnostic workflow. Do not assume any specific repository structure — every project this runs in may have a different stack, folder convention, and spec-generating agent. Everything this workflow "knows" about where specs live comes from (a) active discovery in the current repository and (b) a list of known patterns (see the table in Step 1), used as a hint, never as an assumed truth.

## Why this matters

AI agents (Kiro, Spec Kit, Copilot workspace, Cursor, Claude Code, etc.) generate requirements/design/tasks documents that become the basis for implementation. These documents can have gaps, contradictions between each other (e.g. design doesn't cover a requirement, tasks implement something design never anticipated), or ambiguous points only a human can decide. This workflow exists to catch that **before** code gets written, and to produce a reviewable artifact that can be handed back to the agent that generated the spec for correction.

## Workflow

### Step 1 — Discover SPECs in the current repository

Do not assume which agent generated specs in this project. Discover it:

1. Check the table of known folder conventions below (Kiro, Spec Kit, etc.) as a starting point for the Glob search — not a closed list.
2. Run Glob/Grep for:
   - Known patterns from the table below (e.g. `**/.kiro/specs/*/`, `**/.specify/specs/*/` or similar).
   - Generic heuristic: any directory that contains **two or more** of these file families together: `requirements*.md`, `design*.md`, `plan*.md`, `tasks*.md`, `spec*.md`, `todo*.md` (case-insensitive). This catches uncatalogued conventions (teams that created their own `specs/`, `docs/specs/`, `.ai/specs/` folder, etc.).
   - Common root directories even without an agent marker: `specs/`, `.specs/`, `docs/specs/`.
3. For each spec found, build a record with: relative path, inferred "agent" (marker folder name, e.g. `.kiro` → `kiro`; if no recognizable marker, use `generic`), list of files present, and the most recent modification date among those files.
4. Ignore dependency/build directories (`node_modules`, `.git`, `vendor`, `dist`, `build`, `.terraform`) during the search.

If nothing is found, tell the user clearly and ask whether the spec lives somewhere unconventional (they may provide a path directly).

#### Known agent spec folder patterns

This table is a starting point to speed up the search — **not a closed list**. Agent conventions change over time and teams customize their own structures — the generic heuristic above (two or more `requirements*/design*/plan*/tasks*/spec*` file families together in a directory) is what guarantees coverage beyond this table. Always confirm by actually inspecting the current repository; never assume a pattern listed here exists just because it's catalogued.

| Agent / tool | Typical marker directory | Typical files per spec |
|---|---|---|
| Kiro (AWS) | `.kiro/specs/<slug>/` | `requirements.md`, `design.md`, `tasks.md`, `.config.kiro` (JSON with `specId`, `workflowType`, `specType`) |
| GitHub Spec Kit (`specify` CLI) | `.specify/` (config/memory) + `specs/<NNN-slug>/` | `spec.md`, `plan.md`, `tasks.md`, and optionally `research.md`, `data-model.md`, `contracts/`, `quickstart.md` |
| Generic ad hoc convention | `specs/`, `.specs/`, `docs/specs/`, `.ai/specs/` | varies — usually some subset of `requirements.md`/`design.md`/`plan.md`/`tasks.md` |
| Cursor | `.cursor/rules/*.mdc` | Not a "spec" convention — these are project rules, not feature specs. Do not treat as a spec unless the team created its own separate specs folder. |
| Windsurf | `.windsurf/rules/*` | Same caveat as Cursor — rules, not feature specs. |
| Cline | `.clinerules`, `cline_docs/` | No fixed structured-spec convention; if one exists, it likely falls under the generic heuristic. |
| Claude Code (teams practicing spec-driven dev with Claude) | `.claude/specs/<slug>/` or `specs/<slug>/` | Varies by team — no official product convention; search via the generic heuristic. |

How to use this table:

1. Run Glob for whichever marker directory above seems worth trying (no need to try all of them every time — start with the most likely given what already exists in the repo, e.g. a `.kiro/` at the root is a strong signal for Kiro).
2. Regardless of what this table suggests, also run the generic heuristic (`requirements*/design*/plan*/tasks*/spec*` files grouped in the same directory) — that's the primary coverage mechanism, this table is just a search optimization.
3. When naming the "agent" in the output path (`.ai/spec-reviews/<agent>/...`), use the marker folder name without the leading dot (`.kiro` → `kiro`, `.specify` → `specify`) when recognizable; otherwise use `generic`.

### Step 2 — Determine what has already been reviewed

This workflow stores its reviews inside the project itself, under `.ai/spec-reviews/<agent>/<spec-slug>/`, never outside the repository (so the history travels with the project and can be versioned like the rest of the code).

For each spec discovered in Step 1:

1. Check whether `.ai/spec-reviews/<agent>/<spec-slug>/` exists with at least one `review-*.md`.
2. If it exists, read the front matter of the most recent review (highest timestamp in the filename) — it records a "fingerprint" per source file (see the review template in Step 6). Recompute the current fingerprint of each source file:
   - Prefer `sha256sum <file>` (available via Git Bash on any platform). If unavailable, use `git hash-object <file>` (if the file is tracked) or size+mtime as a last resort.
3. Classify the spec into one of these states:
   - **Never reviewed** — no `review-*.md` found.
   - **Changed since last review** — a review exists, but at least one fingerprint changed.
   - **Review pending a decision** — a review exists, fingerprints match, but the front matter shows `status: awaiting-human-decision` (i.e. open questions remained in the last round).
   - **Approved** — a review exists, fingerprints match, `status: approved-no-remarks`.

Build a table for the human operator with: spec name, agent, path, state, and date of last review (if any). Make it clear that "Never reviewed", "Changed since last review", and "Review pending a decision" are candidates for a new review round; "Approved" needs no action unless the user asks for it.

### Step 3 — The operator chooses what to review

- If there are 4 or fewer pending specs, use the interactive question tool (single-select) listing each one with a short description (agent + state), for a quick decision.
- If there are more than 4, present the full table as text and ask the user to name which one(s) to review.
- If the user already indicated the spec directly in the original request (e.g. "review spec X"), skip the listing and go straight to Step 4 — but still confirm the spec exists and show its current state (never reviewed / changed / etc.) before proceeding.

### Step 4 — Read and synthesize the chosen SPEC

Read **all** files that make up the spec (requirements, design, tasks, agent config files like `.config.kiro`, notes, ADRs, diagram images if they have an associated textual description — anything inside the spec folder that isn't clearly irrelevant). Don't limit yourself to "obvious" filenames: read the entire directory.

Produce an objective synthesis (not a copy of the documents) answering:
- What exactly is being proposed for implementation?
- Which components/modules/systems are touched?
- What is the "done" criterion according to the documents themselves?

This synthesis becomes section 1 of the review file (Step 6).

#### Step 4.1 — Gather additional project context (outside the spec folder)

The spec folder alone rarely tells the whole story. Before moving to gap/contradiction analysis, look for documentation already existing elsewhere in the repository that helps understand the real context of the project — this is what lets you detect, for example, a spec that contradicts an already-made architectural decision, or that reinvents something already documented elsewhere. Look for (without limiting yourself to):

- `README.md` at the root and in subfolders relevant to the components mentioned by the spec.
- AI agent instruction files (`CLAUDE.md`, `AGENTS.md`, `.cursor/rules/*`, `.windsurf/rules/*` or equivalents) — these usually document conventions and constraints the spec should respect.
- Architecture/decision documentation (`docs/`, `ARCHITECTURE.md`, `adr/` or `docs/adr/`, `CONTRIBUTING.md`).
- Other specs already present under the same marker directory (e.g. other folders inside `.kiro/specs/`), which may reveal already-established patterns or dependencies between specs.
- Central config/infra files the spec references (e.g. CI pipeline, schema, API contracts) when the spec talks about them without attaching them.

You don't need to read the whole repository — prioritize documents that mention the same components, modules, or glossary terms as the spec. Treat what you find here as **supporting context**, not part of the spec itself: use it to back findings in Step 5 (e.g. "spec X contradicts what's documented in `docs/architecture.md`, section Y") and to enrich the synthesis, but you don't need to reproduce this material in the review file beyond what's necessary to justify a finding.

### Step 5 — Analyze gaps, contradictions, and ambiguities

Treat this as a cross-audit between the documents, not an isolated reading of each one. Specifically check:

**Gaps**
- Requirements with no corresponding coverage in the design.
- Design or tasks that depend on something no requirement mentions (scope appearing out of nowhere).
- Error cases, empty/null input, concurrency, permissions/security, rollback, observability — if relevant to the spec's domain and not mentioned.
- Acceptance criteria that aren't testable/verifiable (vague language: "should work well", "adequate", "when necessary" without defining when).

**Contradictions**
- Two statements (in the same document or across documents) that can't both be true.
- Tasks that implement something different from what the design specified for the same point.
- Inconsistent terminology for the same concept (may indicate divergent understanding, not just naming).
- Contradiction with the additional context gathered in Step 4.1 — e.g. the spec proposes something that conflicts with a convention documented in `CLAUDE.md`/`AGENTS.md`, with an already-recorded architectural decision, or with another already-approved spec in the same project.

**Ambiguities / unclear cases**
- Decisions marked "TBD", or implicit but never explicitly stated.
- Assumptions about external dependencies (APIs, variables, services) the spec doesn't confirm exist.
- Multiple equally valid implementation paths the spec doesn't choose between.

Each finding in these three categories must become an **objective question to the human operator**, accompanied by 2–3 concrete suggested paths forward (not just "fix this"). Reference the specific requirement/section whenever possible, so the question is traceable back to the original document — this is what lets the agent that generated the spec (or another one) correct it precisely afterward.

### Step 6 — Present to the operator and generate the review file

1. Present an executive summary in chat: short spec synthesis + finding count by category (gaps/contradictions/ambiguities).
2. For the most critical findings with clearly distinct paths (at most 3–4), you may use the interactive question tool to let the operator decide right there — record the chosen answer in the review file as an "Operator decision". The remaining findings stay as open questions in the file, to be answered asynchronously or forwarded to the original agent.
3. Use the review template below, filling in the synthesis, the file map, and each finding under the Gaps/Contradictions/Ambiguities sections, including the current fingerprint of each source file in the front matter.
4. Save to `.ai/spec-reviews/<agent>/<spec-slug>/review-<YYYYMMDD-HHMMSSmmm>.md`, where `mmm` is 3-digit milliseconds (use the system's real current date/time, don't make it up). This precision avoids filename collisions when multiple reviews are generated in quick succession. Create the directories if they don't exist.
5. Set `status` in the front matter:
   - `awaiting-human-decision` if questions remain without a recorded decision.
   - `awaiting-agent-correction` if the operator already decided everything and the next step is for the original agent to adjust the documents.
   - `approved-no-remarks` only if the analysis found no relevant gaps, contradictions, or ambiguities (rare case — don't force artificial findings just to fill the section, but also don't mark as approved if the search wasn't actually thorough).

```markdown
---

spec_agent: <kiro | specify | generic | other>
spec_name: <spec slug/name>

reviewer: spec-review skill (Claude Code)
source_files:
  - path: <e.g. requirements.md>
    fingerprint: <sha256 or git-hash-object or size+mtime>
  - path: <e.g. design.md>
    fingerprint: <...>
  - path: <e.g. tasks.md>
    fingerprint: <...>
status: awaiting-human-decision
open_questions_count: <N>
---

# SPEC Review: <spec name>

## 1. Summary of the proposed implementation

<Objective natural-language synthesis of what the spec proposes to implement — don't copy the documents, synthesize. Include: what will be built, which components/modules are touched, and what the "done" criterion is according to the documents themselves.>

## 2. Files analyzed

| File | Role in the spec |
|---|---|
| <file> | <e.g. functional requirements in EARS format> |

## 3. Additional context considered

<Short list of documents outside the spec folder that were consulted (README, CLAUDE.md/AGENTS.md, architecture docs, other specs, etc.) and what each contributed to the analysis. If nothing relevant was found beyond the spec itself, state this explicitly.>

## 4. Gaps identified

### G1. <short gap title>

- **Description:** <what's missing and why it matters>

### C1. <short contradiction title>

- **Description:** <what conflicts with what>
- **Question to the operator:** <objective question>

  1. <option 1>
  2. <option 2>
- **Operator decision:** _(fill in once decided)_

<Repeat C2, C3... or state there are no contradictions.>

## 6. Ambiguities / not fully clarified cases

### A1. <short title>
- **Where:** <requirement/section>
- **Description:** <what's ambiguous or implicit>
- **Question to the operator:** <objective question>
- **Suggested paths forward:**
  1. <option 1>
  2. <option 2>
- **Operator decision:** _(fill in once decided)_

## 7. Well-defined points (optional)

<If relevant, highlight aspects of the spec that are clear and complete — this keeps whoever corrects the spec from touching what's already good.>

## 8. Next steps

- Forward this file to the agent responsible for the original spec (`<spec_agent>`), referencing the G/C/A items above for document correction.
- After the correction, run the `spec-review` workflow again — it will automatically detect the change in source files (fingerprint) and treat this spec as pending a new review.
```

### Step 7 — Close the loop

Tell the user the path of the generated file and explain explicitly that it can be forwarded to the agent that generated the original spec (Kiro, Copilot, Cursor, etc.) to correct the documents based on the recorded questions and decisions. Suggest running this workflow again after the correction — Step 2 will automatically detect that the source files changed (different fingerprint) and treat it as pending a new review.

## Limits and cautions

- Never invent a spec structure that doesn't exist in the current repository — always discover it first (Step 1).
- Don't treat the known-patterns table as exhaustive; it exists to speed up the search, and the generic heuristic is what guarantees coverage in projects with uncatalogued conventions.
- Don't generate artificial findings just to have content — if a spec is genuinely well-written and complete, say so.
- The review file is the durable artifact (goes into the project's version control); the chat conversation is just the interactive surface for quick decisions.
