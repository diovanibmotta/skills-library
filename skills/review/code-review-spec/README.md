# code-review-spec

Reviews already-implemented code against the SPEC that originated it (requirements/design/tasks) and against the spec review generated by the `spec-review` skill, plus applies a generic quality/security/best-practices check over the changed files.

## Trigger

```
/code-review-spec
```

## What it does

1. Locates the originating spec and, if it exists, its `spec-review` review (`.ai/spec-reviews/`) — treating recorded "operator decisions" as the source of truth over the raw spec text
2. Synthesizes the spec into a list of verifiable acceptance criteria
3. Gathers additional project context (`README.md`, `CLAUDE.md`/`AGENTS.md`, architecture docs, CI config) to calibrate what "best practice" means in this specific repository
4. Asks the user for the diff base, then splits the changed files into implementation code, spec files, and config/infra/CI
5. Builds a traceability matrix mapping every spec criterion to what the diff actually delivered (implemented / partial / not implemented / divergent)
6. Runs a stack-agnostic quality and security check — actually executing the project's own lint/test/security tools when available
7. Classifies every finding as Blocking / Important / Suggestion and generates a versioned review file at `.ai/code-review/<spec_agent>/<spec_slug>/review-<timestamp>.md`

## Target stack

Any project, any language, any AI agent that implemented the code (Kiro, Copilot, Cursor, Claude Code, or a human developer)

## Usage

```
/code-review-spec
```

## Notes

- Pairs with the `spec-review` skill: that one audits the spec *before* implementation, this one audits the *code* afterward using the spec (and its review) as acceptance criteria
- Review files are saved inside the project (`.ai/code-review/`) so history is versioned alongside the code
- Re-running after fixes automatically detects changed implementation files via fingerprint
- Never infers what can be verified by actually running the project's lint/security/test tooling
