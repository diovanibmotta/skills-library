# Implementation Code Review Against a SPEC

Project- and LLM-agnostic workflow. Do not assume any specific repository structure, language, or toolchain — discover everything by actively inspecting the current repository. This workflow is the natural continuation of `spec-review`: that one audits the spec **before** implementation; this one audits the **code** after it's written, using the spec (and the spec review, if it exists) as the acceptance criteria.

## Why this matters

A well-reviewed spec doesn't guarantee the implementation followed what was decided — the implementing agent may have misread a requirement, ignored a decision the operator recorded during the spec review, cut a corner on an error case, or introduced a quality/security problem unrelated to the spec itself. This workflow exists to close that loop: systematically compare what was asked for (spec + review decisions) against what was actually delivered (the diff), and complement it with a best-practices check that's independent of the spec.

## Workflow

### Step 1 — Locate the spec and, if it exists, the spec review

Don't assume where the spec lives. Discover it:

1. Look for artifacts generated by the `spec-review` skill at `.ai/spec-reviews/<agent>/<slug>/review-*.md`. If there's more than one review for different specs, ask the user which spec is being reviewed this round (unless the original request already makes it clear, e.g. from the branch name or an explicit ask).
2. If a review is found, read its front matter and body: `spec_path`, `spec_agent`, `spec_name`, `source_files` (with fingerprint), `status`, and especially every **"Operator decision"** recorded in the Gaps/Contradictions/Ambiguities sections. These decisions are the source of truth when the original spec was ambiguous — treat them as an effective part of the specification, taking priority over the original text when they conflict.
3. If `status` is `awaiting-human-decision` (open questions still pending in the spec review), flag this clearly to the user before proceeding: any choice the implementer made for a point not yet decided deserves extra scrutiny in this code review, because there was no firm acceptance criterion when the code was written.
4. If no spec review is found for what's being implemented, tell the user and ask whether they want to:
   - Proceed anyway, comparing the code only against the raw spec (without cross-referencing decisions); or
   - Run the `spec-review` skill first against the corresponding spec.
   Don't block the flow — just make it explicit in the final result that no spec review existed at the time of this analysis.
5. If no review is found, locate the raw spec using the same heuristic as the `spec-review` skill (known marker directories like `.kiro/specs/`, `.specify/`/`specs/`, or the generic heuristic: two or more `requirements*/design*/plan*/tasks*/spec*` file families together in a directory). Spec discovery itself doesn't need to be duplicated here beyond what's necessary to find the same directory `spec-review` would use.

### Step 2 — Read and synthesize the spec as acceptance criteria

Read all spec files (requirements, design, tasks, ADRs, notes). Produce an objective list of verifiable acceptance criteria, each traceable to a specific section/requirement/task:

- What was supposed to be implemented (behavior, endpoints, resources, business rules).
- Which tasks/steps the spec defined as scope (if there's a `tasks.md` or equivalent).
- Explicit "done" criteria in the spec.
- Operator decisions (Step 1.2) that override or complete ambiguous points in the original spec.

This list becomes the basis for the traceability matrix in Step 4 — don't skip this synthesis even if the spec looks simple, because it's what prevents comparing the code against a superficial reading.

### Step 2.1 — Gather additional project context (outside the spec)

The spec and the diff alone rarely tell the whole story of "what's expected in this project". Before building the traceability matrix and the quality check, look for documentation already existing in the repository that helps correctly interpret both the spec and the code — this is what lets you recognize, for example, that a change follows (or breaks) an already-established architectural convention, even if the spec doesn't mention it explicitly. Look for (without limiting yourself to):

- `README.md` at the root and in subfolders relevant to the components touched by the change.
- AI agent instruction files (`CLAUDE.md`, `AGENTS.md`, `.cursor/rules/*`, `.windsurf/rules/*` or equivalents) — these usually document code conventions, tagging/naming rules, mandatory quality gates, and architectural constraints the implementation should respect even if the spec doesn't repeat them.
- Architecture/decision documentation (`docs/`, `ARCHITECTURE.md`, `adr/` or `docs/adr/`, `CONTRIBUTING.md`).
- CI/CD configuration (`.gitlab-ci.yml`, `.github/workflows/*`, `Jenkinsfile` or equivalent) — reveals which lint/security/test gates are already formally required in the project, which serves as the minimum floor for Step 5.
- Other specs/reviews already present in the same marker directory, which may reveal established patterns or dependencies between features.
- Central config files referenced by the spec or the diff (schema, API contract, routes file) when they help confirm the implementation is consistent with the rest of the system.

You don't need to read the entire repository — prioritize documents that mention the same components, modules, or terms as the spec, or that are touched by the diff. Treat what you find here as **supporting context**: use it to back findings in Steps 4 and 5 (e.g. "the implementation doesn't follow the tagging convention documented in `CLAUDE.md`", or "the module's `README.md` already describes this flow differently, suggesting an inconsistency not covered by the spec") and to calibrate what counts as "best practice" *in this specific project*. If nothing relevant is found beyond the spec itself, state this explicitly in the final file instead of omitting the section.

### Step 3 — Determine the scope of files changed by the implementation

Never assume the comparison baseline. **Always ask the user** which ref/commit/branch the diff should be computed against (e.g. `git diff <base>...HEAD`, a specific commit, or the branch behind an MR/PR) — don't infer a default yourself, even if it seems obvious from the repository history.

Once the base is confirmed:

1. Run `git diff --stat <base>...HEAD` (or equivalent) to list changed, added, and removed files.
2. Split the files into:
   - **Implementation code** — what will actually be audited in Steps 4 and 5.
   - **Spec/spec-review files** (`.ai/spec-reviews/`, the spec folder) — if these show up in the diff, treat it as a signal the spec was changed during implementation; record this separately as a finding (it can be legitimate — the spec evolving alongside the code — but it needs to be visible in the review).
   - **Config/infra/CI files** — relevant both for traceability (did the spec ask for an infra change?) and for the quality check in Step 5.
3. Ignore dependency/build/vendor directories (`node_modules`, `.git`, `vendor`, `dist`, `build`, `.terraform`, lock-file caches when irrelevant to the functional diff).
4. For each implementation code file, get the full diff (not just `--stat`) — the analysis in Steps 4 and 5 needs the actual content of the changes, not just the list of names.

### Step 4 — Build the spec → implementation traceability matrix

For each acceptance criterion from the Step 2 synthesis (requirement, task, operator decision), check the diff:

| Criterion | Status | Evidence | Note |
|---|---|---|---|
| <requirement/task/decision> | Implemented / Partial / Not implemented / Divergent | <file:line or diff excerpt> | <note> |

Pay special attention to:

- **Requirements with no evidence in the diff** — possible implementation gap (unless code already existed covering it before; verify before marking as missing).
- **Diff changes with no corresponding requirement/task** — not necessarily a problem (could be a needed refactor, a bug fix found along the way, etc.), but should be listed for visibility — especially if it's a structurally significant change.
- **Approach divergence** — the design specified one approach (e.g. a synchronous endpoint) and the implementation did something different (e.g. async via queue) without this being recorded as an operator decision. This is a divergence to be escalated, not silently accepted.
- **Operator decisions (Step 1.2) ignored** — the most critical point of this workflow: if the spec review recorded an explicit decision and the code doesn't follow it, this is always a finding, regardless of apparently low severity.
- **Error/edge cases the spec (or domain common sense) required** — empty/null input, permissions, concurrency, rollback, idempotency — when mentioned in the spec but absent from the code.

### Step 5 — Code quality and security check (stack-agnostic)

This step is independent of the spec — it evaluates the changed code on its own merits. Use the checklist below to cover the dimensions worth checking (correctness, error handling, security, readability/duplication, tests, performance, observability), but **adapt it to the repository's actual stack**:

1. Use the context already gathered in Step 2.1 (`CLAUDE.md`, `AGENTS.md`, `README.md`, `CONTRIBUTING.md`, CI config) — they define what "best practices" means *in this specific repository* (e.g. a Terraform project has `tflint`/`tfsec`/`checkov` as the minimum standard; a Node project might have `eslint`/`prettier`). Treat these documented gates as a mandatory floor, not a suggestion.
2. If the project's lint/security/test tools are available in the current environment, **actually run them** on the changed files (or the whole project/module, if the tool doesn't support partial scope) and fold the real result into the finding — don't infer by reading what can be verified by direct execution.
3. For what isn't covered by an automated tool, evaluate by reading: naming and clarity, avoidable duplication, error handling, vulnerability potential (injection, hardcoded secrets, missing input validation, access control), test coverage for new/changed behavior, and any obvious performance red flag in the context of the domain.
4. Don't invent generic quality rules that conflict with a convention already established in the project (e.g. don't suggest a different naming pattern than what the repository already uses consistently) — the goal is adherence to what the project itself already practices, plus universal security/correctness best practices.

#### Quality dimensions checklist

This checklist is a starting point for Step 5, **not a closed list or a substitute for the current project's conventions**. Always prioritize what the repository's `CLAUDE.md`/`AGENTS.md`/`CONTRIBUTING.md`/CI config already documents as standard — this checklist fills in what isn't covered by the project's own convention.

- **Correctness** — the code does what the requirement/task asked, with no unintended side effects. Obvious domain edge cases (empty/null input, numeric limits, no network/timeout, concurrency) are handled when plausible in context. Conditional logic covers all relevant branches; no dead or unreachable paths that should be reachable.
- **Error handling** — errors are caught/handled at the appropriate level, not silently swallowed. Error messages are useful for diagnosis (without leaking sensitive information). Partial failures (e.g. one of several operations fails) have defined behavior — rollback, retry, or explicit failure, as appropriate to the domain.
- **Security** — external inputs (user, API, file, environment variable) are validated/sanitized before use in queries, shell commands, file paths, or templates. No hardcoded secrets, tokens, or credentials in code or versioned examples. Access control/permissions consistent with what the spec or the project's standard requires (e.g. no endpoint/resource exposed without the expected authorization check). New dependencies introduced have no already-established alternative in the project without justification (avoids duplicated responsibility and unnecessary attack surface). Logs don't record sensitive data (passwords, tokens, PII) in plain text.
- **Readability and duplication** — variable/function/module names communicate intent, consistent with the vocabulary already used in the project. No avoidable duplication of logic that already exists elsewhere in the changed code or the repository. Complexity of the change is proportional to the problem (no premature abstraction, no generic solution for a single case).
- **Tests** — new or changed behavior has corresponding test coverage, following the framework/pattern already used in the project. Relevant edge cases identified in the traceability matrix (Step 4) have tests, not just the happy path. Changed/removed tests have a clear justification (not just "the test broke and got deleted").
- **Performance** — no obvious performance red flag introduced by the change (e.g. N+1 query, unnecessary loop over a large collection, avoidable recomputation) — evaluate only what's plausible in the domain context, without speculating about hypothetical scale the spec never mentioned.
- **Observability** — changes with relevant operational impact (new critical flow, external integration, async job) have minimal logging/metrics for production diagnosis, when this pattern already exists elsewhere in the project.
- **Documentation** — changes that alter public behavior (API, CLI, config) update the corresponding documentation (README, CHANGELOG, interface comment) when the project already maintains that kind of documentation.

### Step 6 — Classify findings by severity

Each finding (from Steps 4 and 5) gets a severity:

- **Blocking** — breaks an explicit spec/decision requirement, introduces a real security vulnerability, or a clear correctness bug. Must be resolved before merge.
- **Important** — relevant non-critical divergence, error/edge-case handling gap, missing test for new behavior, significant technical debt introduced by the change itself.
- **Suggestion** — readability/structure improvement that doesn't block, simplification opportunity, style nit.

Don't inflate the list with artificial findings just to fill sections — if a dimension is genuinely good, state that explicitly instead of forcing a weak "Suggestion".

### Step 7 — Generate the code review file

1. Use the review template below, filled in with: the spec synthesis (Step 2), the additional project context considered (Step 2.1), the traceability matrix (Step 4), the quality/security findings (Step 5) already classified by severity (Step 6), and the diff base used (Step 3).
2. Save to `.ai/code-review/<spec_agent>/<spec_slug>/review-<YYYYMMDD-HHMMSSmmm>.md` (same naming convention as the `spec-review` skill, with 3-digit milliseconds to avoid collisions — use the system's real current date/time). If there's no identifiable `spec_agent`/`spec_slug` (no spec/spec-review found), use `generic/<slug-derived-from-diff-or-branch>`. Create the directories if they don't exist.
3. In the front matter, record: the spec and spec review used as input (with their fingerprints, if available), the diff base (`<base>...<head>`, resolved to a real commit hash), the fingerprint of each reviewed implementation file, and the finding count by severity.
4. Set `status` in the front matter:
   - `changes-needed` if there's at least one Blocking or Important finding.
   - `approved-with-suggestions` if there are only Suggestion-severity findings.
   - `approved-no-remarks` if no relevant finding was found (rare case — don't force this just to close the loop).

### Step 8 — Present to the operator and close the loop

1. In chat, present an executive summary: how many spec criteria were covered vs. missing/divergent, finding count by severity, and whether there were pending spec-review decisions at the time of implementation.
2. Point to the path of the generated file and explain it can be forwarded to the agent that implemented the code (or used directly by the developer) to fix the Blocking/Important items.
3. Suggest running this workflow again after the adjustments — since the implementation file fingerprints and the diff base are recorded in the front matter, a new round can detect what changed since the last review.

## Limits and cautions

- Never invent a spec or project structure that doesn't exist — discover it by actually inspecting the repository (Step 1).
- The "Operator decisions" recorded by `spec-review` take priority over the original spec text when they conflict — but this doesn't excuse recording the divergence if the implementer also didn't follow the decision.
- Actually running lint/security/test tools (when available) is always preferable to inferring by reading — don't skip this for convenience.
- Don't generate artificial findings just to fill sections; if the implementation is genuinely adherent and good quality, say so clearly.
- The file under `.ai/code-review/` is the durable artifact (versioned with the project); the chat conversation is just the interactive surface.
